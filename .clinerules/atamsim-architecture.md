## Brief overview
This rule set captures the key architectural conventions of the `atamsim` package — a domain-specific extension that implements the Architecture Tradeoff Analysis Method (ATAM) on top of the TinyTroupe framework. Any code changes, phase implementations, extractor additions, or stakeholder template additions should respect these structural principles.

## Package positioning and dependency rule
- `atamsim` is a **sibling package** to `tinytroupe/`, living at the repository root. It is **not** a submodule of `tinytroupe` and must **never** modify core TinyTroupe abstractions.
- Dependencies flow in one direction: `atamsim` → `tinytroupe` → foundation. `atamsim` imports from `tinytroupe` but `tinytroupe` never imports from `atamsim`.
- Prefer lazy imports (inside functions/methods) for TinyTroupe symbols that are only needed at runtime, not at module load time — e.g., `from tinytroupe.utils import fresh_id` inside `models._generate_id`.
- The package must remain importable even when TinyTroupe's runtime dependencies (LLM clients, LlamaIndex) are absent. The pure-data layer (`models.py`) achieves this by falling back gracefully.

## ATAM → TinyTroupe abstraction mapping
The package maps ATAM domain concepts onto TinyTroupe core classes. These mappings are fixed and should not be reinterpreted:

| ATAM Concept | TinyTroupe Base | atamsim Implementation |
|---|---|---|
| Evaluation session | `TinyWorld` | `ATAMSession` (in `session/atam_session.py`) |
| Stakeholder | `TinyPerson` | Generated by `ATAMStakeholderFactory` from templates |
| Stakeholder panel factory | `TinyPersonFactory` | `ATAMStakeholderFactory` (in `stakeholders/stakeholder_factory.py`) |

- `ATAMSession` subclasses `TinyWorld` and inherits `run()`, `broadcast()`, parallel execution, display, and serialization.
- `ATAMStakeholderFactory` subclasses `TinyPersonFactory` and calls `super().__init__(context=...)` with a merged context string.
- Stakeholder agents are plain `TinyPerson` instances — there is **no** `ATAMStakeholder` subclass of `TinyPerson`. Role identity is carried by persona attributes and the `occupation` persona override.

## Phase orchestration pattern
- Phases are **plain Python objects**, not simulation entities. They must **not** subclass `TinyPerson` or `TinyWorld`.
- All phases inherit from the abstract base `ATAMPhaseBase` (in `phases/base_phase.py`) and implement three abstract methods: `prepare_stimulus()`, `execute()`, `extract_results()`.
- The concrete helper `ATAMPhaseBase._broadcast_and_run(stimulus, n_steps)` delegates to `session.broadcast()` and `session.run()`. Phase implementations should use this helper rather than calling broadcast/run directly.
- `ATAMSession` owns a lazy cache of phase instances (`_phase_instances`) keyed by `ATAMPhase` enum. Phase dispatch goes through `ATAMSession.run_phase()` / `_get_phase_instance()`.
- Phase outputs are stored in `ATAMSession.phase_results` (keyed by `ATAMPhase` enum value string) and accumulated artifacts go into the session's `scenarios`, `concerns`, and `approaches` lists.
- New phases should be added to the `ATAMPhase` enum in `models.py` **before** creating the phase class. The canonical phase order is: `PRESENTATION`, `APPROACH_IDENTIFICATION`, `SCENARIO_GENERATION`, `SCENARIO_PRIORITIZATION`, `APPROACH_ANALYSIS`, `CONCERN_IDENTIFICATION`, `BRAINSTORMING`.

## Domain model layer (`models.py`)
- `models.py` is the foundational data layer. It defines **enums**, **dataclasses**, **Pydantic models**, and **conversion helpers**. It has no hard dependency on TinyTroupe at import time (only a lazy import of `fresh_id`).
- **Enums**: `QualityAttribute` (11 values: availability, modifiability, performance, security, testability, usability, scalability, reliability, deployability, cost), `ConcernType` (4 values: risk, non_risk, sensitivity_point, tradeoff), `ATAMPhase` (7 values).
- **Dataclasses** represent the domain artifacts: `Scenario` (stimulus → environment → artifact → response → response_measure), `Concern`, `ArchitecturalApproach`, `ATAMReport`, `StakeholderRoleTemplate`.
- **Pydantic models** are used as `response_format` for structured LLM output: `ScenarioItemModel`, `ScenariosExtractionModel`, `ConsolidatedScenariosModel`, `PriorityVoteModel`, `ConcernItemModel`, `ConcernsExtractionModel`, `ApproachItemModel`, `ApproachesExtractionModel`.
- **Conversion helpers** (`scenario_item_to_dataclass`, `concern_item_to_dataclass`, `approach_item_to_dataclass`, `parse_quality_attribute`, `parse_concern_type`) transform Pydantic instances into dataclass instances. All string-to-enum parsing functions have **sensible defaults** (e.g., `parse_quality_attribute` defaults to `PERFORMANCE`) and accept common aliases/synonyms.
- ID generation uses `_generate_id(prefix)` which wraps `tinytroupe.utils.fresh_id`. It returns a formatted string like `"scenario_3"`. It falls back to a UUID-based string when TinyTroupe is not importable.

## Extraction by composition (not inheritance)
- Extractors (`ScenarioExtractor`, `ConcernExtractor`, `VoteExtractor`) **wrap or call** `tinytroupe.extraction.ResultsExtractor` but do **not** subclass it.
- Each extractor defines a domain-specific Pydantic model (from `models.py`) to pass as `response_format` to the LLM, then converts the parsed result into domain dataclasses using the conversion helpers.
- Extractors take a `prompt_template_path` pointing to a Mustache template in `prompts/`. The default template path is bundled with the package.
- `ATAMReportGenerator` (in `extraction/report_generator.py`) aggregates accumulated session artifacts into an `ATAMReport` and is called by `ATAMSession.generate_report()`. It does not make LLM calls itself — it assembles already-extracted data.
- New extraction needs should follow the same pattern: define a Pydantic model in `models.py`, create an extractor class in `extraction/`, add a prompt template in `prompts/`, and a conversion helper if needed.

## Stakeholder template system
- Role templates are instances of `StakeholderRoleTemplate` defined in `stakeholders/templates.py`. There are **10 predefined templates**: Project Sponsor, System Architect, Lead Developer, Security Expert, DevOps Lead, End User Representative, Product Manager, QA Lead, Data Architect, Project Manager.
- The `ALL_TEMPLATES` list and `TEMPLATES_BY_NAME` dict are the canonical registries. New roles must be added to both.
- Each template specifies: `role_name`, `description`, `responsibilities`, `quality_priorities` (list of `QualityAttribute`), `concerns`, and optional `persona_overrides` (dict merged into the generated persona, e.g., `{"occupation": "...", "attitude": "..."}`).
- `ATAMStakeholderFactory.create_from_template()` generates a single agent from a template. `create_panel()` generates multiple. `create_custom_stakeholder()` creates a role without a template.
- When `ENABLE_GROUNDING_DOCUMENTS` is `True` and an `architecture_docs_path` is provided, generated stakeholders should receive a `FilesAndWebGroundingFaculty` so they can `LIST_DOCUMENTS` and `CONSULT` architecture artifacts.

## Configuration separation
- `atamsim` uses its **own** `ATAMConfig` class (in `config.py`), reading from `atamsim/config.ini`. This is **intentionally separate** from TinyTroupe's `config_manager`.
- `ATAMConfig` reads only the `[ATAM]` section. It never touches LLM model settings, API keys, or any TinyTroupe configuration.
- Config keys: `DEFAULT_STEPS_PER_PHASE` (int, default 5), `SCENARIO_MAX_PER_STAKEHOLDER` (int, default 5), `PRIORITIZATION_VOTE_RANGE` (int, default 10), `ENABLE_GROUNDING_DOCUMENTS` (bool, default true), `CONSOLIDATION_ENABLED` (bool, default true).
- Access via the module-level singleton: `from atamsim.config import config; config.get("KEY")` or the typed helpers `config.get_int("KEY")` / `config.get_bool("KEY")`.
- New ATAM config parameters must be added to both `config.py` (`_DEFAULTS` dict) and `atamsim/config.ini` (`[ATAM]` section with a comment).

## Prompt templates
- LLM prompts live in `prompts/` as Mustache templates (`.mustache` files).
- Extractors and phases reference these templates by relative path. The default path for each extractor should point to the bundled template.
- When adding a new prompt, follow the naming convention: `<phase_or_extractor_name>.mustache` (e.g., `scenario_generation.mustache`, `concern_extraction.mustache`).

## Naming and file conventions
- Module names use `snake_case` (e.g., `atam_session.py`, `scenario_extractor.py`, `base_phase.py`).
- Class names use `PascalCase`. Core session/factory/config/report classes are prefixed with `ATAM` (e.g., `ATAMSession`, `ATAMStakeholderFactory`, `ATAMConfig`, `ATAMReport`, `ATAMReportGenerator`, `ATAMPhaseBase`). Domain entity and extractor classes do **not** carry the prefix (e.g., `Scenario`, `Concern`, `ScenarioExtractor`, `ConcernExtractor`).
- Enum members use `UPPER_SNAKE_CASE` with lowercase string values (e.g., `QualityAttribute.PERFORMANCE = "performance"`).
- Configuration keys use `UPPER_SNAKE_CASE` in `config.ini` and are accessed by the same string via `config.get()`.
- Tests live in `atamsim/tests/` and follow the `test_<module>.py` pattern. The conftest provides fixtures that do not require LLM access.

## Scaffolding and stub convention
- The package was scaffolded with method bodies raising `NotImplementedError` and descriptive messages referencing the scaffolding plan phase. When implementing a stub, replace the `raise NotImplementedError(...)` with the actual logic and remove the "not yet implemented" wording from docstrings.
- Stubs should always have complete type annotations, full docstrings (Args/Returns/Raises), and correct signatures — only the body is deferred.
- Unit tests in `atamsim/tests/` cover the implemented pure-data layer (`models.py`). When implementing a stub, add or update the corresponding test file.
