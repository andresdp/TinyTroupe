## Brief overview
This rule set captures the key architectural conventions of the TinyTroupe project — an LLM-powered multi-agent simulation framework. Any code changes, extensions, or refactors should respect these structural principles.

## Core architecture layers
- The codebase is organized into distinct layers: **Simulation Control** (`control.py`), **Environment** (`environment/`), **Agent Cognition** (`agent/`), **Supporting Subsystems** (factory, tools, enrichment, extraction, validation, experimentation, steering), and **Foundation** (config, clients, utils, logging).
- Dependencies flow downward: user code → control → environment → agent → supporting subsystems → foundation. Avoid upward dependencies or circular references. Cross-layer imports inside functions (lazy imports) are acceptable and already used to break circular dependencies (e.g., `from tinytroupe.agent import TinyPerson` inside methods in `control.py`).

## Transactional caching pattern
- Any state-changing method on `TinyPerson`, `TinyWorld`, or `TinyFactory` that participates in a simulation should be decorated with `@transactional`.
- The decorator captures the function name, args, and kwargs as a hash, checks the `cached_trace`, and either replays the cached state or executes and records the result.
- Parallel execution paths use `@transactional(parallel=True)` combined with `begin_parallel_transactions()` / `end_parallel_transactions()`.
- Function outputs must be encodable via `Transaction._encode_function_output` — supported types are `None`, `TinyPerson`, `TinyWorld`, `TinyFactory`, lists of those, and JSON-serializable scalars/containers. Do not return unsupported types from transactional methods.

## Configuration management
- All runtime parameters flow through the global `config_manager` instance of `ConfigManager`, initialized from `config.ini`.
- Use `config_manager.get("key")` or the `get_config("key", override_value)` helper for reading config values. Do not read `config.ini` directly in subsystem code.
- When a function parameter should default to a config value, use `None` as the default and apply the `@config_manager.config_defaults(key="config_key")` decorator rather than reading config at definition time.
- New configuration parameters must be added to both `ConfigManager._initialize_from_config()` and documented in `config.ini`.
- The `[OpenAI]` section governs model and API settings; `[Simulation]` governs execution behavior; `[Cognition]` governs memory; `[ActionGenerator]` governs quality checks; `[Logging]` governs log output.

## Agent cognition model
- Agents use a dual-memory system: **episodic memory** (verbatim log) and **semantic memory** (consolidated embeddings via LlamaIndex).
- Memory consolidation is configurable and should remain optional (`ENABLE_MEMORY_CONSOLIDATION`).
- Agent actions are generated by `ActionGenerator`, which uses LLM calls with structured JSON schema output (`CognitiveActionModel` / `CognitiveActionsModel`).
- When extending agent capabilities, prefer adding **mental faculties** or **tools** rather than modifying the core `TinyPerson` class.

## LLM client abstraction
- All LLM calls go through the `client()` factory in `clients/`, never directly to the OpenAI/Azure/Ollama SDK.
- New backends must implement the `send_message(messages, **kwargs)` interface returning `{"role": ..., "content": ...}`.
- Structured output uses Pydantic response models passed as `response_format` to `send_message`.
- API call caching (`CACHE_API_CALLS`) and concurrency limiting (`MAX_CONCURRENT_MODEL_CALLS`) are handled at the client layer — do not implement retry or throttling logic in calling code.

## Environment and execution
- Agents always operate inside a `TinyWorld` (or subclass). The world manages `step()` / `run(n)` turn-taking, stimulus broadcasting via `broadcast()`, and display.
- Parallel agent actions within a step are controlled by `PARALLEL_AGENT_ACTIONS` and use the parallel transaction mechanism for cache safety.
- New environment types should subclass `TinyWorld` and override `step()` logic rather than reimplementing stimulus routing.

## Factory pattern for agent generation
- Use `TinyPersonFactory` with a context description to generate agent panels. Generation runs in parallel when `PARALLEL_AGENT_GENERATION` is enabled.
- Factories enforce name and persona uniqueness across the simulation.
- Custom factories should subclass `TinyFactory` and be registered with the simulation via `add_factory()`.

## Tool design
- Tools subclass `TinyTool` and must implement `actions_definitions_prompt()`, `actions_constraints_prompt()`, and `_process_action(agent, action)`.
- Tools can declare `real_world_side_effects=True` to trigger safety warnings.
- Tools support optional `ArtifactExporter` and `Enricher` for output handling.
- Tool dispatch is handled by the agent's action processing loop; tools do not invoke LLM calls directly unless they use the `client()` abstraction.

## Serialization and state management
- All simulation entities (`TinyPerson`, `TinyWorld`, `TinyFactory`, tools) inherit from `JsonSerializableRegistry`, which provides `encode_complete_state()` / `decode_complete_state()`.
- The `Simulation` class manages `cached_trace` (persisted) and `execution_trace` (current run) as lists of `(prev_hash, event_hash, output, state)` tuples.
- Parallel segments store outputs in a dict keyed by event hash at a single trace position.
- State encoding captures agents, environments, and factories by name; decoding resolves them back via `name_to_agent` / `name_to_environment` / `name_to_factory` registries.

## Quality control and validation
- `ActionGenerator` optionally runs proposition-based quality checks (persona adherence, self-consistency, fluency, suitability, similarity) scored 0–9 by the LLM.
- `QUALITY_THRESHOLD` determines pass/fail; failures trigger regeneration or direct correction.
- Propositions are defined in `validation/propositions.py` and can be reused for post-hoc validation via `TinyPersonValidator`.

## Profiling and extraction
- `Profiler` analyzes agent populations with demographics, persona composition facets (with LLM-driven normalization via `Normalizer`), correlations, and visualizations.
- `ResultsExtractor` converts simulation transcripts into structured data (CSV/JSON).
- Both store raw DataFrames for programmatic reuse.

## Naming and file conventions
- Package and module names use `snake_case` (e.g., `tiny_person.py`, `action_generator.py`).
- Class names use `PascalCase` prefixed with `Tiny` for core abstractions (e.g., `TinyPerson`, `TinyWorld`, `TinyTool`, `TinyFactory`).
- Configuration keys use `UPPER_SNAKE_CASE` in `config.ini` and `lowercase` in the `_config` dict.
- Example notebooks live in `examples/`; research artifacts in `publications/`; tests in `tests/`.
